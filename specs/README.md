# Supplementary Specs

本目录用于承接 `SPEC.md` 的实现级补充规格。

原则：

- `SPEC.md` 负责总范围、总体约束、阶段结论。
- 本目录下的文档只补“会影响开发拆分和接口落地”的硬细节。
- 不继续扩写新的大总纲，避免文档层级失控。

当前建议保留 3 份补充规格 + 4 份执行文档，已经足够支撑首期开发：

## 1. `AUTH-COMMENT-SPEC.md`

用途：

- 锁定 `GitHub OAuth` 与 `Telegram Login` 的登录流程
- 锁定评论提交、审核、删除、封禁权限
- 锁定会话 cookie、风控和审计边界

适用人群：

- 后端接口实现
- 评论区前端实现
- 管理审核页实现

## 2. `CONTENT-SCHEMA-SPEC.md`

用途：

- 锁定 `content/articles` 与 `content/downloads` 的 frontmatter 结构
- 锁定下载条目中 `versions / files / links` 的字段规则
- 锁定哪些字段会参与页面渲染与搜索提取

适用人群：

- 内容 schema 实现
- 下载页实现
- 内容校验脚本实现

## 3. `SEARCH-SPEC.md`

用途：

- 锁定站内搜索的数据来源、索引规则、排序规则
- 锁定分页协议与返回结构
- 锁定首页搜索入口与搜索结果页行为

适用人群：

- 搜索 API 实现
- 搜索结果页前端实现
- 内容索引构建实现

## 4. 现在不建议继续拆的文档

以下文档暂时没有必要单独立项，否则会过度设计：

- 单独的部署 SPEC
- 单独的 UI 视觉 SPEC
- 单独的运营后台 SPEC
- 单独的监控告警 SPEC

这些内容目前留在 `SPEC.md` 总纲里已经够用，等首期骨架跑起来再拆。

## 5. 执行文档

### `IMPLEMENTATION-PLAN.md`

用途：

- 把首期开发顺序、依赖关系、阶段完成标准写死
- 避免 Workers 与本地部署并行乱做
- 让任务拆分可以直接落到 issue / 看板

### `API-CHECKLIST.md`

用途：

- 把接口落地时的请求、权限、校验、返回、错误码、测试点收紧
- 避免实现到一半才发现接口风格不统一
- 让前后端联调有明确收口

### `ISSUE-BREAKDOWN.md`

用途：

- 把开发计划拆成可直接创建 issue 的卡片
- 锁定依赖顺序、交付物、验收条件
- 避免不同人并行时重复做或反复返工

### `ENV-SETUP.md`

用途：

- 锁定 Workers 与本地部署两种模式下的环境变量和存放位置
- 给开发、联调、部署提供统一模板
- 避免变量命名漂移、私钥误入仓库、不同环境口径不一致
