# Search Spec

## 1. 范围

本文件补充首期站内搜索实现细节，覆盖：

- 搜索入口
- 搜索 API
- 索引来源
- 排序规则
- 分页协议
- 搜索结果页行为

## 2. 目标

首期搜索必须满足：

- 可搜索文章
- 可搜索下载条目
- 搜索结果支持分页
- 用户可区分文章结果与下载结果
- API 与页面行为在 Workers / 本地部署模式下一致

## 3. 非目标

首期不做：

- 全文模糊搜索引擎集成
- 拼音搜索
- 高级筛选面板
- 搜索历史
- 个性化推荐

## 4. 搜索入口

至少保留两个入口：

- 站点全局头部搜索入口
- 首页搜索入口

搜索结果页路由固定：

- `/search`

URL 参数建议：

- `q`
- `type`
- `page`
- `pageSize`

## 5. 数据来源

### 5.1 文章

最小索引字段：

- `title`
- `summary`
- `tags`

### 5.2 下载条目

最小索引字段：

- `title`
- `summary`
- `tags`
- `versions.version`
- `files.name`

## 6. 索引构建

首期建议采用“构建期提取 + 运行时查询”的轻量方案：

1. 构建期从 `content/articles` 与 `content/downloads` 提取搜索文档
2. 生成统一搜索索引数据
3. 运行时 API 读取索引并执行过滤、排序、分页

适用原因：

- 内容量首期通常不大
- 不需要引入额外搜索服务
- Workers 与本地部署都容易复用

## 7. 查询协议

接口：

- `GET /api/search?q=xxx&type=all|articles|downloads&page=1&pageSize=20`

参数规则：

- `q` 去掉首尾空白后为空时，返回空结果
- `type` 默认 `all`
- `page` 默认 `1`
- `pageSize` 默认 `20`
- `pageSize` 上限建议 `50`

## 8. 返回结构

建议统一返回：

```json
{
  "q": "subtitle",
  "type": "all",
  "page": 1,
  "pageSize": 20,
  "total": 37,
  "totalPages": 2,
  "items": []
}
```

每个结果项至少包含：

- `kind`：`article` 或 `download`
- `title`
- `slug`
- `summary`
- `tags`
- `url`

下载结果可额外返回：

- `matchedVersions`
- `matchedFiles`

## 9. 排序规则

首期固定采用简单可解释排序：

1. 标题精确命中
2. 标题前缀命中
3. 标签命中
4. 摘要命中
5. 下载版本号命中
6. 下载文件名命中

同分情况下可再按：

- `publishedAt` 倒序
- `updatedAt` 倒序

## 10. 分页规则

首期分页为硬要求：

- 搜索结果页必须支持页码切换
- API 必须返回 `total` 与 `totalPages`
- 页码变更时保留 `q` 与 `type`
- 空结果时 `page` 强制回到 `1`

前端行为建议：

- `page=1` 时可省略 URL 参数，也可统一保留
- 翻页时滚动回结果区顶部
- 切换 `type` 时重置到第一页

## 11. 前端行为

### 11.1 搜索输入

- 回车触发搜索
- 点击搜索按钮触发搜索
- 连续输入不要求首期做实时联想

### 11.2 搜索结果页

必须包含：

- 当前关键词展示
- 结果总数
- 类型切换
- 分页控件
- 空结果提示

建议包含：

- 关键词高亮
- 下载结果的版本/文件命中提示

## 12. 性能与缓存

- 索引构建尽量在构建期完成
- 运行时避免每次请求重新扫描 Markdown 原文
- 搜索请求应可被短时缓存
- `q` 为空时不执行重逻辑

## 13. 成功标准

- 搜索结果可稳定覆盖文章与下载条目
- 结果排序符合预期且可解释
- 分页行为正确
- Workers 与本地部署都能复用同一套索引与 API 结构
